# Documentation Workflow

This page outlines a simple docs-as-code workflow for keeping technical documentation clear, current, and aligned with product changes.

## Source of Truth

Documentation pages are maintained as version-controlled Markdown or MDX files in Git.

## Change Intake

Updates can start from engineering notes, release changes, issue tracking, screenshots, or existing reference documents.

## Drafting

New or updated information is reviewed, restructured, and written in a format that is clear for the intended audience.

## Review

Content should be checked for:

- technical accuracy
- clarity
- structure
- consistency with current product behavior

## Release Alignment

Documentation updates should be reviewed alongside product releases so that published guidance matches the shipped system.

## Screenshots and Visual Updates

Screenshots should be refreshed when the interface changes, especially for task flows, settings, and support instructions.

## Deployment Flow

After review, approved changes are committed to Git and merged into the production branch.

The documentation site is then rebuilt and redeployed automatically.
